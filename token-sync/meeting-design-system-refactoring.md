## 배경

기존 디자인 시스템 구조의 한계로 인해 전면 리팩토링을 진행하기로 결정.
신규 Figma 파일을 기준으로 토큰 → 컴포넌트 → 패턴 순으로 정리하며,
최종적으로 스토리북으로 이관하여 전사 공유 목적으로 활용.

## 1. 토큰 구조 변경

### 결정 사항

| 항목 | 기존 | 변경 |
|------|------|------|
| 레이어 수 | primitive / semantic / component (3단계) | primitive / semantic (2단계) |
| 정의 순서 | primitive 먼저 → semantic 순 | **semantic 먼저 → primitive 순** |

### 변경 이유

primitive를 먼저 고정해두면 디자인 결정이 그 범위 안으로 제한되는 문제가 있었음.
semantic(의미와 용도)을 먼저 정의한 뒤 그에 맞는 raw 값을 채우는 방식으로 전환.

### semantic 정의 방향 (디자이너 초안 기준)

Figma 노트 내용: *"html, css 코드베이스 응용해서 단순화하자"*

토큰 카테고리:
```
레이아웃    패딩 / 마진 / 갭 / 정렬
스타일      백그라운드 / 보더 / 래디우스 / 타이포그라피 / 이펙트
```

> **미결 사항**: shape vs radius 카테고리 명칭 결정 필요
> CSS에는 shape 개념이 없고 border-radius만 존재.
> "래디우스로 퉁칠지, 쉐입으로 카테고리를 나눌지" 논의 중.
> → 퍼블리셔 의견: CSS 기준으로 `radius`로 통일 권장

- [ ] semantic 토큰 네이밍/종류 최종 정의 (디자이너)
- [ ] primitive 정의 (semantic 확정 후)
- [ ] shape vs radius 결론 내기

## 2. 진행 방식

### Figma 작업

- 기존 파일 수정 아닌 **신규 파일 생성**으로 진행
- 토큰 정리 완료 후 퍼블리셔에게 1차 공유
- 이후 컴포넌트 리팩토링 순으로 진행

### 퍼블리셔 작업

토큰 공유받은 후 Tailwind 유틸리티 CSS로 재정리.

```
Figma 신규 토큰 확정
    ↓ 퍼블리셔에게 공유
vars.css 재작성 (semantic 기준)
    ↓
tailwind.css @theme 재정의
    ↓
영향받는 컴포넌트 CVA 수정
```

## 3. 토큰 변경 영향도 분석

### ⚠️ component 레이어 제거 영향

기존 component 토큰(버튼 높이, 인풋 패딩 등)이 사라지면
현재 CVA에서 참조하던 값들을 직접 Tailwind 유틸리티로 대체해야 함.

| 영향 항목 | 현재 방식 | 변경 후 |
|---------|---------|--------|
| 버튼 높이/패딩 | component 토큰 참조 | semantic 토큰 or Tailwind 유틸리티 직접 사용 |
| 인풋 높이/패딩 | component 토큰 참조 | 동일 |
| 태그/뱃지 radius | component 토큰 참조 | semantic `radius/*` 직접 참조 |
| 탭 아이콘 크기 | component 토큰 참조 | semantic `size/icon/*` 직접 참조 |

### ⚠️ 네이밍 변경 시 영향 범위

`vars.css` 전면 재작성이 필요하고, 아래 파일들이 연쇄적으로 영향받음.

```
vars.css          전면 재작성
tailwind.css      @theme 변수명 전면 수정
ui/*/index.ts     CVA 클래스명 수정 (bg-*, text-*, border-* 등)
markup/components 일부 인라인 클래스 수정
index.css         @layer components 내 변수 참조 수정
```

> **권장**: 신규 vars.css 작성 후 컴포넌트 파일을 하나씩 마이그레이션.
> 한 번에 전부 바꾸지 말고 컴포넌트 단위로 순차 적용.

## 4. 컴포넌트 리팩토링 계획

### 구조 및 명칭

기존 "소형 컴포넌트" → **코어 컴포넌트**로 명칭 변경.

Figma 기준 컴포넌트 카테고리 (Material Design 참고):

| 카테고리 | 컴포넌트 목록 |
|---------|------------|
| 입력 | 버튼, 버튼그룹, 체크박스, 라디오, 스위치, 텍스트필드, 셀렉트 |
| 데이터디스플레이 | 아바타, 뱃지(카운트), 칩(태그), 디바이더, 아이콘, 리스트, 테이블, 툴팁 |
| 피드백 | 배너, 토스트 |
| 서페이스 | 아코디언, 카드, 페이퍼(섀도우/모달) |
| 네비게이션 | 링크, 메뉴, 페이지네이션, 탭 |
| 레이아웃 | 카드, 컨테이너, 그리드, 스택 |

> **서페이스 = z-order 개념** (Figma 노트)
> 섀도우나 overlay 처리가 필요한 컴포넌트 그룹.

### ⚠️ 이슈 — 현재 코드와 명칭 불일치

신규 컴포넌트 명칭과 기존 코드 컴포넌트명이 달라 혼선 가능성 있음.

| Figma 신규 명칭 | 현재 코드 |
|--------------|---------|
| 텍스트필드 | InputBase, InputGroup |
| 칩(태그) | Badge (variant: hold 등) |
| 셀렉트 | Select + Combobox (2개) |
| 페이퍼 | Dialog / Sheet |
| 스택 | 없음 (신규) |
| 그리드 | 없음 (신규) |

- [ ] 코드 컴포넌트명과 Figma 컴포넌트명 매핑 테이블 작성 (퍼블리셔)
- [ ] Stack, Grid 컴포넌트 신규 개발 여부 결정

### 진행 순서

```
1단계  코어 컴포넌트 (입력/데이터디스플레이/피드백/서페이스/네비게이션)
2단계  섹션 단위 컴포넌트 패턴 정리
3단계  레이아웃 / 페이지 단위
```

## 5. UI 패턴 파일 — 네이밍 제안

기존 섹션 단위 컴포넌트를 패턴별로 묶어 신규 파일에서 관리.
여러 부서가 "우리 서비스에 어떤 UI가 있는지" 파악하는 용도.

### 추천 이름

| 이름 | 설명 | 추천 여부 |
|------|------|---------|
| **UI 패턴북** | 패턴을 모아둔 북 형식, 직관적 | ✅ 추천 |
| **UI 컴포지션** | 컴포넌트 조합 단위임을 명확히 표현 | ✅ 추천 |
| 서비스 UI 라이브러리 | 범위가 넓어 모호할 수 있음 | 보통 |
| 패턴 플레이북 | 영문 혼용, 직관성 낮음 | 비추 |

> **최종 의견**: `UI 패턴북` 또는 `UI 컴포지션` 중 선택 권장.
> Figma 파일명 예시: `[AON] UI 패턴북` / `[AON] UI 컴포지션`

## 6. 스토리북 이관 계획

### 목적

기획자 · 개발자 · 디자이너가 다음을 파악할 수 있는 공유 공간:
- 현재 시스템에 어떤 UI 컴포넌트가 있는지
- 어떤 패턴의 화면 구성이 가능한지

### 관리 기준

`markup/components` 폴더 구조를 스토리북 스토리 기준으로 활용.

```
markup/components/
├── inputs/       → Input 카테고리 stories
├── table/        → 테이블 패턴 stories
├── tabs/         → 탭 패턴 stories
├── modal/        → 서페이스 카테고리 stories
├── layout/       → 레이아웃 패턴 stories
└── ...
```

### 스토리 구성 아이디어

| 스토리 레벨 | 내용 | 대상 |
|-----------|------|------|
| **코어** | 개별 컴포넌트 variant/size | 디자이너·퍼블리셔 |
| **패턴** | 컴포넌트 조합 (폼, 테이블+검색 등) | 퍼블리셔·개발자 |
| **템플릿** | 화면 레이아웃 단위 (1단/2단/마스터디테일) | **기획자·개발자** |

> 기획자에게는 **템플릿 레벨** 스토리만 공유 링크 제공 권장.
> 컴포넌트 단위까지 보여주면 오히려 혼란 가능성 있음.

### 배포

외부 접근 가능하도록 **GitHub Pages 또는 Vercel** 배포 예정.

- [ ] 패턴/템플릿 stories 작성
