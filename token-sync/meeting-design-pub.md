## 안건 6. 디자인 ↔ 퍼블 싱크 방식

> 현재 디자인과 퍼블리싱 작업이 각자 진행되면서 서로의 변경 사항을 인지하지 못하는 경우가 발생하고 있습니다.
> 아래 두 가지 흐름에 대해 각각 기준을 정해두어야 합니다.

### 6-1. 디자인 → 퍼블 방향

디자이너가 Figma에서 컴포넌트나 토큰을 변경했을 때 퍼블리셔에게 전달되는 방식입니다.

현재 문제는 **디자이너가 Figma를 수정해도 퍼블리셔가 알 수 없다**는 것입니다.
변경 내용이 말로 전달되더라도 무엇이 바뀌었는지 정확히 파악하기 어렵고, 누락이 발생할 수 있습니다.

제안하는 방식:
- Figma에서 변경 시 **해당 컴포넌트/토큰에 코멘트를 남기는 것을 규칙**으로 정합니다.
- 코멘트 형식 예: `[변경] Button secondary 배경색 #EFF7FF → #E0F0FF / 사유: 대비비 개선`
- 코멘트가 달리면 퍼블리셔가 확인 후 코드에 반영하고, 반영 완료 시 코멘트에 답글로 완료 표시합니다.

> 추후 자동화 방향: Figma 토큰 변경 → AI가 변경사항 감지 → vars.css 자동 업데이트

### 6-2. 퍼블 → 디자인 방향

퍼블리셔가 구현 과정에서 디자인과 다르게 처리했거나, 디자인에 없는 케이스를 추가했을 때 디자이너에게 전달되는 방식입니다.

현재 문제는 **퍼블에서 변경한 내용이 디자인 파일에 반영되지 않아 디자인과 실제 화면이 달라지는 것**입니다.
시간이 지날수록 두 파일의 괴리가 커지고, 신규 화면 작업 시 잘못된 디자인을 기준으로 작업하게 됩니다.

제안하는 방식:
- 코드에서 디자인과 다르게 처리한 부분은 **작업 완료 후 Figma 해당 노드에 코멘트**로 내용을 남깁니다.
- 코멘트 형식 예: `[퍼블 변경] Input height 32px → 40px 적용 / 사유: 터치 영역 확보`
- 디자이너는 코멘트를 확인하고 Figma를 실제 구현 기준으로 업데이트합니다.

> 추후 자동화 방향: git commit 시 변경된 컴포넌트 파일 감지 → Figma 해당 노드에 코멘트 자동 등록

### 6-3. Single Source of Truth 기준

디자인과 퍼블이 서로를 참조할 때 **어느 파일이 최종 기준인지** 명확히 해야 합니다.

| 항목 | 기준 파일 후보 | 결정 |
|------|-------------|------|
| 컴포넌트 디자인 | Figma | |
| 토큰 값 | Figma (AON Design Tokens) | |
| 실제 구현 스펙 | 코드 (vars.css, CVA) | |
| 화면 레이아웃 | Figma or 코드 | |

> 일반적으로 **디자인 결정은 Figma**, **구현 세부사항은 코드**가 기준이 됩니다.
> 단, 두 파일이 다를 경우 어느 쪽을 따를지 규칙을 미리 정해두어야 혼선이 없습니다.

## 안건 7. 디자인 토큰 변경 가이드

> 퍼블 소스와 연결된 토큰은 변경 범위에 따라 영향도가 다릅니다.
> 디자이너가 토큰을 수정하기 전에 아래 기준을 확인하고 진행해야 합니다.

### 7-1. 절대 변경 금지 — 퍼블리셔 동의 필수

**이름 변경**은 종류에 상관없이 반드시 퍼블리셔에게 먼저 공유해야 합니다.
Figma에서 토큰 이름이 바뀌면 `vars.css`의 CSS 변수명도 함께 수정해야 하며,
누락 시 해당 변수를 사용하는 모든 컴포넌트의 스타일이 깨집니다.

> **이름 변경 = 반드시 퍼블리셔 선동의 후 진행**

---

### 7-2. 값 변경 시 위험 — 레이아웃 파괴 가능

색상이 아닌 **크기·간격 계열** 토큰은 값이 바뀌면 화면 레이아웃이 틀어집니다.
변경 전 반드시 퍼블리셔와 협의가 필요합니다.

| 토큰 | 현재 값 | 위험한 이유 |
|------|--------|-----------|
| `size/height/sm` | 24px | 버튼, 인풋, 뱃지 높이 직접 참조 |
| `size/height/md` | 32px | 전체 폼 요소 기본 높이 |
| `size/height/lg` | 40px | 대형 컴포넌트 높이, 테이블 행 높이 |
| `spacing/xs ~ xxl` | 4~32px | 컴포넌트 내부 패딩 전체에 사용 |
| `font/size/xs ~ xl` | 12~24px | 텍스트 크기 변경 시 줄바꿈·잘림 발생 |
| `font/lineHeight` | 1.3 / 1.4 / 1.5 | 멀티라인 텍스트 컨테이너 높이에 영향 |
| `border/width/default` | 1px | 레이아웃 계산에 포함됨 |

---

### 7-3. 비교적 자유롭게 변경 가능

**색상 값**은 변경해도 레이아웃은 유지됩니다.
단, 아래 사항은 변경 후 확인이 필요합니다.

| 토큰 | 주의사항 |
|------|---------|
| `color/primary/default` | 텍스트에도 쓰여서 배경과 명도 대비 확인 필요 |
| `color/text/disabled` | 너무 연하게 바꾸면 WCAG 접근성 기준 미달 가능 |
| `color/border/focus` | 포커스 표시 역할, 주색과 구분되어야 함 |
| `color/danger/*` | error 상태 전체에 사용, 빨간 계열 유지 필요 |

아래 토큰은 비교적 자유롭게 조정 가능합니다.

```
color/neutral/*   gray 팔레트 미세 조정
color/*/light     배경용 연한 색 조정
radius/sm, md     모서리 둥글기 (레이아웃 영향 없음)
shadow/*          그림자 강도 조정
```
